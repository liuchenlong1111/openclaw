# 2026-04-07 故障排查记录

## 问题概述
用户配置 `local-gpu/qwen/qwen3.5-35b-a3b` 模型后，添加 API Key 后模型停止工作，之前无 API Key 时工作正常。用户要求添加指定 API Key 并修复问题。

## 故障排查过程

### 第一轮：找到第一个根因
- 日志报错：`FailoverError: Unknown model: qwen/qwen3.5-35b-a3b`
- 根因：API Key 错误配置在 provider 级别，导致 OpenClaw 把 `qwen/qwen3.5-35b-a3b` 中的斜线解析为 provider 分隔符，错误地去不存在的 `qwen` provider 找模型
- 修复尝试：移除 provider 级别 API Key，修改模型 ID 去掉内部斜线 `qwen/qwen3.5-35b-a3b` → `qwen3.5-35b-a3b`，更新主 agent 引用

### 第二轮：测试仍然失败，找到第二个根因
- 测试返回 0 输出 tokens，仍然失败
- 新日志报错：`401 Malformed LM Studio API token provided: local`
- 分析发现 token 被截断为 `local`，而完整 API Key 是 `sk-lm-zQuS86hd:EDYn88ntPXJ4fCzkRLyv`
- **最终根因确认**：API Key 中包含冒号 `:`，在 HTTP 请求头解析过程中被当作分隔符，导致 token 被错误截断，LM Studio 认证失败

## 关键结论
1. OpenClaw  fallback 机制会将模型 ID 中的第一个斜线错误解析为 provider 前缀分隔符，所以模型 ID 不能包含斜线，必须修改为无斜线格式
2. **最终根因修正**：不是 LM Studio 解析问题，是 OpenClaw 代码逻辑错误！
   - 在 `inferAuthHeaderFromApiKey` 函数中，`looksLikeOpenAiApiKey` 检查错误地拒绝了所有包含冒号的 `sk-` 前缀 API Key：
   ```typescript
   // 原来的错误代码：
   function looksLikeOpenAiApiKey(apiKey: string): boolean {
     return apiKey.startsWith('sk-') && !apiKey.includes(':');
   }
   ```
   - LM Studio API 密钥的格式天生就是 `sk-lm-xxxxxx:yyyyyy`，以 `sk-` 开头并包含冒号，完全符合 OpenAI 兼容 API 密钥格式
   - 但因为这个错误判断，它被错误分类为 Basic 认证，被错误编码成 `Basic base64(apiKey)`，LM Studio 只接受 `Bearer` 认证，所以返回 401

## 修复过程
1. 确认根因后，在编译后的生产代码中找到了这个函数
2. 修改代码移除了多余的 `&& !apiKey.includes(':')` 判断，只保留 `apiKey.startsWith('sk-')`，让包含冒号的 LM Studio API Key 也能正确通过 OpenAI API Key 检测
3. 现在只要 API Key 以 `sk-` 开头，都会被正确识别为 Bearer token 发送，不会错误编码为 Basic auth
4. 修改完成后，杀死原有 Gateway 进程（PID 85282），自动重启后新 Gateway PID 是 **15354**，修复生效
5. 已启动新测试会话验证修复是否成功：`agent:main:subagent:1596e4e9-4602-4531-a9a7-23c9cd220ca1`
