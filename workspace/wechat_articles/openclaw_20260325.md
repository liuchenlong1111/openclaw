# OpenClaw 每日资讯：设备配对问题全解析

大家好！今天龙龙仔继续为大家带来 OpenClaw 的实用技术分享。最近在使用 OpenClaw 过程中，不少开发者遇到了设备配对的问题，今天我们就来彻底梳理一下如何解决 `pairing required` 这个常见问题。

## 什么是设备配对？

OpenClaw 设计了安全的设备配对机制，确保只有经过授权的设备才能连接到你的 Gateway 服务。这类似于家里的门锁，只有配对过的"钥匙"才能开门，保护你的本地服务不被未授权访问。

当你看到 `pairing required` 错误提示时，说明 CLI 需要得到 Gateway 的批准才能建立连接。别担心，这不是 bug，而是安全机制在起作用。

## 标准解决流程

### 第一步：打开 Dashboard

首先，在浏览器中访问你的 Gateway 控制面板：

```
http://127.0.0.1:18789/
```

如果你是远程部署，把 `127.0.0.1` 换成你的服务器 IP 地址即可。

### 第二步：找到配对管理入口

在 Dashboard 中，你可以在以下位置找到配对选项：

- **"Auth"** 标签页
- **"Authentication"** 认证菜单
- **"Pairing"** / **"设备配对"**
- 或者在 **"Settings"** → **"Security"** 安全设置中

### 第三步：生成配对码

进入配对管理界面后，点击 **"New Pairing"** 或 **"Generate Code"** 生成一个新的配对码。系统会给你一个 6 位数的数字代码。

### 第四步：在 CLI 中批准配对

拿到配对码后，回到终端运行：

```bash
openclaw pairing approve 你的配对码
```

比如：`openclaw pairing approve 123456`

### 第五步：验证连接

配对完成后，测试一下：

```bash
openclaw gateway status
```

如果不再提示 `pairing required`，说明配对成功了！🎉

## 手动配对方案

如果 Dashboard 找不到配对选项，没关系，我们可以直接通过 CLI 来处理：

```bash
# 首先查看当前有哪些配对请求
openclaw pairing list

# 如果看到你的请求处于 pending 状态，批准它
openclaw pairing approve <request-id>
```

## 应急方案：配置文件直接配对

如果以上方法都不行，可以试试这个应急方案，通过修改配置来解决：

```bash
# 1. 先停止 Gateway 服务
openclaw gateway stop

# 2. 临时禁用认证
openclaw config unset gateway.auth

# 3. 重启 Gateway
openclaw gateway start

# 4. 在 Dashboard 中重新配置认证
# 5. 配置完成后记得重新启用 token 认证
```

> ⚠️ **重要提醒**：禁用认证后，任何人都可以连接你的 Gateway，请记得尽快重新配置认证，不要长期处于开放状态。

## 常见问题排查

### Q: 为什么我已经配对了，还提示需要配对？

A: 可能是 Gateway 重启后配对信息丢失，或者你的 IP 地址发生了变化。重新走一遍配对流程一般就能解决。

### Q: 远程服务器部署，无法访问 127.0.0.1:18789 怎么办？

A: 可以通过 SSH 端口转发把远程端口映射到本地：

```bash
ssh -L 18789:127.0.0.1:18789 your-user@your-server
```

然后就可以在本地浏览器访问 `http://127.0.0.1:18789` 了。

### Q: 有没有办法关闭配对认证？

A: 技术上可以关闭，但**强烈不建议**这么做。尤其是暴露在公网的服务器，开启认证是保护你自己的第一道防线。

设备配对是 OpenClaw 安全机制的重要组成部分，虽然第一次配置可能会遇到一些小麻烦，但为了安全还是值得的。记住这个流程：

1. 🖥️ 打开 Dashboard
2. 🔍 找到配对管理
3. 🔢 生成配对码
4. ✅ CLI 批准配对
5. 🧪 验证连接
